# Accepted Bookings System - Quick Reference

## 🎯 What Changed

### booking_request.html (Owner Side)
**Function:** `acceptRequest(requestId, name, email)`

**Before:** Saved to localStorage only
**Now:** Saves to localStorage AND database

**Key Changes:**
```javascript
// Save to database FIRST
await fetch(`${API_URL}/api/booking/update`, {
    body: JSON.stringify({ 
        status: 'accepted',        // ← Most important
        acceptedAt: timestamp,
        responseDate: timestamp,
        name, email, user_id, property_id, property_name
    })
});

// Then create chat
await createChatRoom(...);
```

**API Endpoint:** `PUT /api/booking/update`
**Required Fields:** `bookingId`, `status: 'accepted'`

---

### websitechat.html (Tenant Side)
**Function:** `loadChats()`

**Before:** Fetched all bookings and showed pending/accepted
**Now:** ONLY shows accepted bookings from database

**Key Changes:**
```javascript
// Filter ONLY accepted from DB
const acceptedBookings = allBookings.filter(b => {
    const status = (b.status || '').toLowerCase();
    return ['accepted', 'accepted_by_owner'].includes(status);
});

console.log(`Accepted: ${acceptedBookings.length}`);

// Render only these
acceptedBookings.forEach(booking => renderChat(booking));
```

**API Endpoint:** `GET /api/booking/requests?user_id={id}`
**Filter:** `status: 'accepted'` or `status: 'accepted_by_owner'`

---

## 🔄 Flow in 3 Steps

```
1️⃣ OWNER ACCEPTS
   ↓
   booking_request.html
   acceptRequest() function
   → Updates DB: status='accepted'
   
2️⃣ DATABASE STORES
   ↓
   MongoDB/Database
   booking_requests collection
   → Document has status='accepted'
   
3️⃣ TENANT SEES IT
   ↓
   websitechat.html
   loadChats() function
   → Filters for status='accepted'
   → Displays in chat list
```

## 📊 Database Schema

**Collection:** `booking_requests` or `bookings`

```javascript
{
    _id: ObjectId,
    user_id: "tenant_id",
    owner_id: "owner_id",
    property_id: "property_id",
    property_name: "Apartment Name",
    name: "Tenant Name",
    email: "tenant@email.com",
    status: "accepted",           // ← FILTER BY THIS
    responseDate: ISO_8601_TIME,
    acceptedAt: ISO_8601_TIME,
    rent: 15000,
    location: "City Name",
    created_at: ISO_8601_TIME,
    updated_at: ISO_8601_TIME
}
```

**Status Values:**
- `"pending"` - Awaiting owner
- `"accepted"` ✅ - Owner accepted (SHOWS IN CHAT)
- `"accepted_by_owner"` ✅ - Alternative name
- `"confirmed"` ✅ - Confirmed
- `"rejected"` - Owner declined (NOT shown)

```
✅ booking_request.html

   ├─ Update localStorage
```

```
✅ website/websitechat.html
   ↓ Page loads
   ↓ loadChats()
   ├─ GET /api/booking/requests
   ├─ Filter: status === 'accepted'
   └─ Render accepted bookings
   ↓

   ↓
✅ Click to open chat
   ↓
✅ Start messaging with owner
```

---

## 🎯 Key Points

1. **Database is source of truth**
   - Status must be `'accepted'` in DB to show in chat

2. **Both endpoints must match**
   - Owner saves with status='accepted'
   - Tenant filters by status='accepted'

3. **User ID must match**
   - Owner saves tenant's user_id
   - Tenant fetches by own user_id

4. **Property details included**
   - Property name shown in chat list
   - Owner name fetched and displayed

5. **Fallback mechanisms**
   - localStorage backup if API fails
   - Alternative endpoints if primary fails
   - Multiple data sources combined
